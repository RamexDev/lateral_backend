Complete REST API reference for the Zwuwur lateral transfer marketplace. This UI is aligned with the current backend implementation and SRS v1.0.

Core Product Rules

  • Matching never crosses bank boundaries.
  • Marketplace access requires a complete profile.
  • Feed matching is live, not persisted.
  • Paywalled fields are returned as "*" until purchased.
  • The frontend is a dumb renderer; the backend decides what is revealed.
  • Purchases are permanent per buyer/target pair and never charged twice.
  • Notifications are Telegram-only.
  • Marketplace responses are cached for 5 minutes.

API Conventions

Content Type

All JSON request and response bodies use application/json; charset=utf-8 unless otherwise stated.

Success Envelope

{
  "success": true,
  "data": {},
  "message": "Optional human-readable message"
}

Error Envelope

{
  "success": false,
  "error": {
    "code": "MACHINE_READABLE_CODE",
    "message": "Human-readable message"
  }
}

Pagination

Param Default Description
page 1 1-indexed page number.
page_size 10 Items per page. Max usually 100. 
{
  "success": true,
  "data": {
    "results": [],
    "page": 1,
    "page_size": 10,
    "total_results": 0
  }
}

Localization

  • Supported languages: en, am.
  • User routes resolve language from the authenticated user's preferred_language.
  • Reference data always has English and Amharic values.

Idempotency

  • Payment webhooks are idempotent by provider charge reference and/or tx_ref.
  • Purchase creation is protected by unique buyer/target constraint and short-lived locks.
  • Repeated webhook delivery must not reveal or charge twice.

Authentication

User API

Bearer User JWT

Used by the Telegram Mini App. Token scope: user.

Admin API

Bearer Staff JWT

Used by the Admin PWA. Token scope: staff.

User Authentication

Mini App requests are authenticated by validating Telegram WebApp initData, then issuing a user JWT. 

Authorization: Bearer <user-jwt>

Admin Authentication

  • Admin PWA users authenticate with email and password.
  • Access token: short-lived JWT.
  • Refresh token: opaque rotating token stored hashed.
  • Roles: super_admin, admin.
Authorization: Bearer <staff-jwt>
Router-token binding User-scope tokens are rejected on /admin/api/v1/*, and staff-scope tokens are rejected on /api/v1/*.

Rate Limits

Area Limit Notes
Feed 60 req/min per user Prevents scraping and excessive refresh.
People 60 req/min per user Same class as feed browsing.
Purchase initiation 10 req/min per user Prevents payment abuse.
Admin login 5 failures / 15 min per IP Lockout after repeated failures.

Business Rules Enforced by API

  • Bank is fixed at registration and cannot be changed.
  • Profile completion requires at least one language version for full name, branch, and neighborhood, plus grade selection.
  • Feed and People require a complete profile.
  • Candidates must be in the same bank.
  • Candidates must be within ±1 grade band.
  • Feed shows candidates interested in the viewer's zone or region.
  • People shows candidates located in the viewer's desired areas.
  • Mutual matches rank first in Feed.
  • Zone-level matches rank above region-broad matches.
  • Contact reveal happens only after confirmed payment.
  • Purchase is permanent per buyer/target pair.

Webhook Design

Payment Webhook Requirements

  • Verify HMAC or provider signature using raw body.
  • Return 200 quickly.
  • Process idempotently.
  • Ignore duplicate success events safely.
  • Validate amount and currency.
  • Mark payment completed only once.
  • Send payment confirmation notification after success.

Recommended Payment Reference Format

zwuwur-payment-{payment_id}

Using the payment ID in the provider reference allows multiple payment attempts for the same purchase while preserving idempotency and preventing double revenue counting.
